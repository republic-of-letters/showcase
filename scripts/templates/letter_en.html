One round, end to end

Read one letter

The fastest way to understand this place is to read one complete exchange. Below is R000, the worked example that ships with the protocol — its data is synthetic on purpose, so it runs anywhere, but its shape is exactly what a real round looks like. When a live project approves a real exchange for display, it will take over this page.

Part 1 — The ask

A question, in a folder, on a branch

A round starts when someone scaffolds a folder and fills in ASK.md. It opens with a machine-readable envelope:

round:    R000
title:    Worked example — mean outcome by group
topic:    T00
kind:     analysis
proposer: "@example-proposer"
data:     []          # a real round lists the tables it touches
Why it matters: before reading a single line of code, the data side knows which topic this serves, what kind of round it is, and its blast radius — exactly which tables it will touch.

Then the ask itself, in prose:

Question. What is the mean outcome y by group, and how does its dispersion compare across groups?

Expected output. A table (group, n, mean_y, sd_y), a bar chart of mean y with a zero reference line, and a one-paragraph reading.
The contract: "Expected output" is the definition of done. The Runner runs until exactly this exists — no more, no less. Ambition lives in the question; scope lives here.
Part 2 — The code

Runnable, bounded, blind to where the data lives

The proposer attaches run.py. It never knows the data's real location — it reads an environment variable the data side sets at run time, and it may write into its own ./result/ folder and nowhere else:

# a real round reads the actual tables instead:
#   DATA_ROOT = Path(os.environ["DATA_ROOT"])
#   df = pl.read_parquet(DATA_ROOT / "<table>.parquet")
Before this code may touch a real dataset, it passes an automated scan — network access, subprocess spawns, file mutation, secret reads, runtime installs all flag it — and then a human on the data side reads it anyway. Then it runs sandboxed: non-root, data read-only, no network. That is the data gate, the second of the three human gates.
Part 3 — The reply

Aggregates come back into the same thread

The data side runs the code and pushes the results — a write-up, figures, small tables — into the same pull request. This one opens with the protocol's loudest rule:

SYNTHETIC — smoke test, do not cite
Why so loud: output produced without real data must be unmistakable, because it once nearly wasn't — an early deployment almost merged unlabelled smoke-test tables as real results. Rules here are scars, not theories. On a real round, the data side's run replaces this placeholder.
Headline. Mean outcome y ranges from −0.031 (charlie) to +0.100 (foxtrot), with near-identical dispersion (sd ≈ 0.15) across all six groups — by construction.
Caveats. Everything: the data is synthesised. This round demonstrates the format only.
The habit to notice: every reply must state what it does not establish. Honest caveats are part of the contract, not a courtesy.
Part 4 — The record

Merged means archived, forever

Continuous checks watch every round: no file over a few MB, no raw-data formats, the folder structure intact. When they're green and the humans on both sides are content, the round merges — and its folder on main becomes the permanent, citable record of the question and its answer. Dead ends merge too, as post-mortems: the archive keeps everything tried, and why it stopped.

That's the whole loop. One question, one folder, one pull request — ask, run, reply, merge. Everything else in the protocol is detail in service of this exchange.

Do it yourself: the onboarding drill has you (and your AI agent) reproduce exactly this round in about 30 minutes — ONBOARDING.md. The original files are at exchange/R000-example-hello.
